【实战记录】大一新生如何从零搭建Hexo 博客并部署到Github (避坑指南)

作为一名计算机系的大一新生，我决定搭建一个属于自己的技术博客，用于沉淀 C++、算法和线性代数的学习笔记。

在对比了多种方案后，我选择了 Hexo (静态博客框架) + GitHub Pages (免费托管) 的经典组合。本文记录了搭建全过程，特别是针对 Windows 环境下各种报错的**“工程化解决方案”**。

一、 环境准备 (Prerequisites)

工欲善其事，必先利其器。搭建 Hexo 需要以下环境：

  1. Node.js:
    • 下载 LTS (长期支持) 版本。
    • 注意：安装时务必勾选 Add to PATH，否则终端无法识别命令。
  2. Git:
    • 下载 Git for Windows。
    • 建议：安装后使用 Git Bash 作为主要终端，避免 PowerShell 的权限问题。
  3. Hexo CLI:
    • 通过 npm 安装：npm install -g hexo-cli

当我尝试运行 hexo s 进行本地预览时，PowerShell 报错：

无法加载文件 hexo.ps1，因为在此系统上禁止运行脚本...

原因分析：Windows 默认的安全策略禁止运行未签名的脚本。
解决方案：放弃 PowerShell，直接在文件夹右键选择 “Git Bash Here”。在 Linux 风格的终端下，一切畅通无阻。

运行 hexo s 后，访问 http://localhost:4000，本地预览成功。

三、 部署到 GitHub (最硬核的部分)

这是最容易踩坑的环节。网上大多数教程推荐使用 hexo-deployer-git 插件，但在 Windows 上，这个插件经常报 Spawn failed 或 Connection timed out。

1. 创建仓库

在 GitHub 新建仓库，名称必须严格遵守格式：

你的用户名.github.io

2. 遇到的第二个坑：自动化插件失效

我尝试配置 _config.yml 并运行 hexo d，结果报错：

Error: Spawn failed
fatal: unable to auto-detect email address

系统无法识别 Git 身份，且插件内部调用 Git 命令时因网络问题崩溃。

3. 终极解决方案：编写 Shell 脚本

既然黑盒插件不稳定，不如直接写一个 Shell 脚本 (deploy.sh)，显式执行 Git 命令。这不仅更稳定，而且逻辑透明。

四、 GitHub Pages 404 问题

推送成功后，访问网址却显示 404。

原因：代码推到了 master 分支，但 GitHub Pages 默认可能在监听 main 分支，或者没有开启 Pages 服务。

解决步骤：

  1. 进入仓库 Settings -> Pages。
  2. 将 Build and deployment 下的 Branch 设置为 master 和 /(root)。
  3. 点击 Save。
  4. 等待 Actions 构建变绿，刷新页面，博客上线！

五、 总结与工作流

这次折腾让我理解了 Git 的底层逻辑：自动化工具再好用，也不如掌握底层命令来得实在。

今后的写作流：

  1. hexo new "文章标题"
  2. 使用 Typora 编写 Markdown。
  3. hexo s 本地预览。
  4. ./deploy.sh 一键发布。
